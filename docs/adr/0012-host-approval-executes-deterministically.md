# Host approval executes a pre-specified diff deterministically

A host approving an Approval is a new inbound event (host action, arriving later). The proposed change was already fully specified when it went to the host (exact new checkout time, exact fee), so approval executes that specified diff against the data store with deterministic code — it does NOT re-enter the LLM pipeline. Re-interpreting a committed action would risk drift and there's nothing left to interpret. The guest notification may use the LLM for friendly wording, but consequential facts (new time, amount charged) fill template slots from the executed change (per ADR-0009). The notification routes to the right guest via the conversation_id the Approval carried (ADR-0005). Denial is symmetric: no mutation, templated "host couldn't accommodate" with optional LLM softening.
