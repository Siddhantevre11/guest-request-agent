# Affirmation binds to a pending item deterministically; never auto-commits

A guest's "yes" / "sounds good" is meaningless on its own — it only resolves against a pending item (an upsell offer or a proposed change) held in memory. We decided: the LLM classifies the message as an affirmation, but deterministic code binds it to the specific pending item by looking up what's actually pending — the LLM never guesses which offer. Affirming an upsell does NOT auto-commit it; consistent with ADR-0007, adding a night is a booking mutation with money attached, so the "yes" converts the offer into a proposed change that routes to the host as an Approval. The host commits.

Ambiguity carve-out: if two items are pending and the guest sends a bare "yes," deterministic code detects the ambiguity and the agent clarifies once ("the extra night, or the late checkout?") rather than guessing. If still unclear after clarifying, it escalates. This is a deliberate narrowing of ADR-0001 (which sends consequential ambiguity straight to escalation): a one-line disambiguation between known pending items is cheaper and better UX than pulling in the host, and it's bounded to a single clarify attempt.
