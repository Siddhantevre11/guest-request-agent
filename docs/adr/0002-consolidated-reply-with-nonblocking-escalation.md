# Consolidated reply, non-blocking escalation, latency budget

A guest message can contain multiple intents that resolve at different speeds; the naive design would either send multiple messages per turn (ordering/race complexity) or wait for the slowest intent (unbounded, if that intent escalates to a human). We decided: one consolidated reply per guest turn, sent within a target ~30s / hard ceiling ~60s, made possible because `escalate()` is fast and non-blocking — it logs the handoff and returns immediately, it does not wait for the human to actually respond. The human's eventual reply is a separate, later follow-up message, not part of this turn's latency budget. This keeps the "wait for every intent to resolve" rule cheap regardless of intent mix, since nothing in the automated turn blocks on a human.
